## Voice Pipeline

Speech Input → VAD → STT → LLM → TTS → Speech Output

---

## Components Explained

### VAD (Voice Activity Detection)
- Detects when the user is speaking  
- Avoids processing silence  
- Prevents unnecessary computation  

---

### STT (Speech to Text)
- Converts audio into text  
- LLM uses this text to understand user input  
- Choose models that support multiple languages & accents  

---

### LLM (Large Language Model)
- Processes the text and generates response  
- Handles:
  - prompts  
  - planning  
  - reasoning  
  - tool integration  

---

### TTS (Text to Speech)
- Converts response into audio  
- Should sound natural and human-like  

---

## Latency Notes

Humans expect responses in **< 500ms** for smooth conversations.

### Typical Latency

| Component | Best Case | Typical |
|----------|----------|--------|
| VAD      | 15–20ms  | 20–80ms |
| STT      | 200–300ms | 400–600ms |
| LLM      | 100–200ms | 500–1000ms |
| Total    | ~415ms   | 1–2s |

👉 Lower latency = better experience

## Why WebRTC?

### Problem with HTTP (TCP)
- Waits for all packets (reliable but slow)  
- Causes delay in real-time applications  

---

### WebRTC (UDP-based)
- Does not wait for lost packets  
- Faster delivery  
- Designed for real-time audio/video  

👉 Better for voice agents  

---

## WebRTC Internals

### Opus Codec
- Compresses audio efficiently  
- Optimized for speech  

---

### Timestamps
- Each packet has a timestamp  
- Ensures correct playback order  

---

### Jitter Buffer
- Handles uneven packet arrival  
- Smooth playback  

---

### Adaptive Bitrate
- Adjusts audio quality based on network  
- Helps avoid lag  

---

### Packet Loss Handling
- Some packets may be lost  
- Audio still continues smoothly  

---

## Final Understanding

WebRTC allows:
- Fast audio delivery  
- Low latency  
- Handling of packet loss and jitter  

👉 Result: smooth, real-time conversation experience
